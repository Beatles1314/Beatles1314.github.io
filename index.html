模块与包

lua的模块与包的定义方法如下：

-- 使用table包装module
utils = {}

-- 模块私有函数
local funtcion private()
    return "hello"
end

-- 模块公有函数
function utils.encode(message)
    return message
end

-- 返回模块
return utils

引入lua模块的方法：

-- 方法一
require('utils')

utils.encode("Hello")

-- 方法二

require 'utils'

对于第三方的模块，可以通过设置LUA_PATH的方法来将库引入。

metatable

metatable主要用于对table的行为进行进一步的扩展，比如实现运算符重载、方法重写等功能。

模式 描述
__add 对应的运算符 ‘+’.
__sub 对应的运算符 ‘-‘.
__mul 对应的运算符 ‘*’.
__div 对应的运算符 ‘/‘.
__mod 对应的运算符 ‘%’.
__unm 对应的运算符 ‘-‘.
__concat 对应的运算符 ‘..’.
__eq 对应的运算符 ‘==’.
__lt 对应的运算符 ‘<’.
__le 对应的运算符 ‘<=’.

例如，重写__toString方法：

mytable = setmetatable({10, 20, 30}, {
    __tostring = function(mytable)
        local message = ''
        for _, value in pairs(mytable) do
            message = message .. ',' .. value
        end
        return message
    end
})
print(mytable)

协程

lua协程的基本方法：

方法 描述
coroutine.create() 创建 coroutine，返回 coroutine， 参数是一个函数，当和 resume 配合使用的时候就唤醒函数调用
coroutine.resume() 重启 coroutine，和 create 配合使用
coroutine.yield() 挂起 coroutine，将 coroutine 设置为挂起状态，这个和 resume 配合使用能有很多有用的效果
coroutine.status() 查看 coroutine 的状态 注：coroutine 的状态有三种：dead，suspended，running，具体什么时候有这样的状态请参考下面的程序
coroutine.wrap（） 创建 coroutine，返回一个函数，一旦你调用这个函数，就进入 coroutine，和 create 功能重复
coroutine.running() 返回正在跑的 coroutine，一个 coroutine 就是一个线程，当使用running的时候，就是返回一个 corouting 的线程号

典型的生产消费模型：


-- 生产者 生产数据后使用yield方法挂起
function producer(count)
    local i = 0
    while true do
        i = i + 1
        coroutine.yield(i)
    end
end

-- 消费者 使用resume接收生产者协程生产的数据
function consumer(co)
    while (true) do
        local state, value = coroutine.resume(co)
        print(value)
    end
end

-- 启动生产者协程
p = coroutine.create( producer, 10 )
-- 启动消费者协程
consumer(p)

文件I/O

lua的文件读写使用io库进行，示例如下：

file = io.open('line.txt', 'w')
-- 读取第一行
print(file:read())
file:close()

-- 读取文件每一行 读取完毕后关闭文件
for line in io.lines('line.txt') do
    print(line)
end

错误处理

lua中使用assert和error来处理错误：

local function add(a, b)
   assert (type(a) == "number", "a不是一个数字") 
   error('抛出了一个异常')
end

pcall接收一个函数及传递给后者的参数，执行后返回函数执行是否有错误：

if pcall(func, xxx) then
    ...
end

xpcall提供一个异常的处理函数，当异常发生时将会调用该函数：

xpcall(功能函数,错误处理函数,功能函数参数)

-- 实例
xpcall(function() print("hello") error("error") end, function() print(debug.traceback()) end)

面向对象

lua中使用table + function来实现面向对象。

User = {
    user_id = '',
    user_name = '',
}


function User:new(obj, user_id, user_name)
    obj = obj or {}
    setmetatable(obj, self)
    self.__index = self
    self.user_id = user_id
    self.user_name = user_name
    return obj
end


function User:say()
    print(self.user_id, self.user_name)
end


user = User:new(nil, '1', 'Mike')
user:say()

高可用、高并发的分布式服务架构

发表于 2020-07-20 更新于 2020-07-21 分类于 web开发

高并发

【参考文献】

高并发的准则是：水平扩展 + 垂直扩展。

垂直扩展

垂直扩展的常见方法：

  1. 提升单机性能，如优化CPU、网卡、硬盘、内存等。
  2. 提升单机架构性能：如使用Cache减少磁盘IO、使用异步框架提升单服务吞吐量、使用无锁数据结构减少响应时间等。

水平扩展

水平扩展主要基于以下架构进行：

反向代理层

通过”DNS”轮询实现，将一个域名配置多个解析ip，每次DNS解析请求来访问dns-server，会轮询返回以上ip。

web应用层

通过nginx实现，当应用层出现瓶颈，通过增加应用节点，在nginx进行配置即可实现。

service层

通过连接池实现，service层出现瓶颈时，通过在新机器上增加service的部署，即可水平扩展。

数据库层

当存在大量请求时，通过对数据库根据业务逻辑进行拆分，将请求分散到不同的库中，以减轻数据库的读写压力。

常用的划分手段为：

1）以范围进行拆分：如，user0存储uid为1-1kw的用户，user1存储uid为1kw-2kw的用户。好处：规则简单，易于处理；数据存储均衡性较好；容易扩展；不足：请求负载不一定均衡，一般来说，新注册用户比老用户活跃，高峰期可能对某个范围的库造成压力。

2）以哈希进行拆分：每一个数据库，根据hash规则存储数据。好处是：请求均衡性较好；不足：不容易扩展，可能需要进行数据迁移。

高可用

【参考链接】

高可用的核心是：冗余 + 故障转移，从架构上需要进行分层，确保各层都保持高可用的状态。

常见互联网分布式架构如上，分为：

1.客户端层：典型调用方是浏览器browser或者手机应用APP。
2.反向代理层：系统入口，反向代理。
3.站点应用层：实现核心应用逻辑，返回html或者json。
4.服务层：如果实现了服务化，就有这一层，即站点应用层不直接访问数据库，service层通过各个数据部门提供封装的对外接口进行数据的操纵。
5.数据-缓存层：缓存加速访问存储。
6.数据-数据库层：数据库固化数据存储。

【客户端 -> 反向代理层】

客户端即是用户发起请求的设备，该过程需要在反向代理层做到冗余及故障转移。

例如，两台nginx反向代理服务器，一台用于线上服务，另外一台作为冗余，自动故障转移使用keepalived存活探测进行。

【反向代理层 -> 站点层】

通过在nginx中配置负载均衡多个冗余站点进行自动故障转移实现该层的高可用。

例如，使用淘宝的nginx_upstream_check_module进行健康检查，配置如下：

upstream name {
    server 192.168.0.21:80;
    server 192.168.0.22:80;
    check interval=3000 rise=2 fall=5 timeout=1000 type=http;
}

【站点层 -> 服务层】

通过冗余服务实现高可用，可以使用服务连接池进行，例如，RPC Client的服务连接池。

【站点层/服务层 -> 数据缓存层】

通过增加数据缓存层的冗余来达到服务的高可用状态，例如，Redis集群模式，当主节点宕机时，启用从节点进行恢复等。

【站点层/服务层 -> 数据库层】

数据库层通过读写分离，主从同步，可以通过实现读库的冗余和写库的冗余进行自动故障转移。

MQ总结

发表于 2020-07-19 更新于 2020-07-27 分类于 web开发

MQ的应用场景

消息队列主要有以下应用场景：

  1. 应用解耦，通过约定好应用间交换数据格式，可以直接通过生产消费的模式进行应用解耦，上游应用无需关注下游应用的实现逻辑，从而进行应用解耦。
  2. 解决异步消息，这也是消息队列主要的应用场景之一。
  3. 流量消峰的问题，可以减少突发并发压力带来的影响。
  4. 日志处理（kafka）。

如何确保消息被发送至RabbitMQ

发送方确认机制：将channel设置为confirm模式，所有在channel发送的消息有一个唯一的ID，一旦消息被投递到队列或写入磁盘后，channel发送给生产者确认消息，如果消息丢失则发送nack信号，通过生产者的回调函数可以处理确认消息。

如何确保消费者消费了消息

接收方消息确认机制：消费者接收每一条信息都必须进行确认，只有消费者确认了消息，消息才会安全地从队列中删除。

如何避免消息重复消费

在生产时，MQ内部针对每条消息生成了一个inner-msg-id，作为去重和幂等的依据；在消费时，可以加上bizId（业务全局唯一ID）作为去重和幂等的条件，进行去重。去重的方式可以采用数据库或者Redis等。

如何解决丢数据的问题

生产者丢数据

  • 事务模式：发送消息前，channel开启事务模式，然后发送消息，失败则进行回滚，成功则提交事务。
  • 确认模式：每条消息被赋予唯一ID，发送成功则给生产者发送ack信号，失败则发送nack信号。

消费者丢数据

采用手动确认模式，消息者消费完成后手动发送ack信号进行确认，如果出现异常导致没有正常ack，则会一直重试发送消息。

队列丢数据

开启队列持久化选项：

  • 队列durable设置为true
  • 发送消息时设置deliveryMode=2

一般配合confirm机制使用。如果rabbitmq挂了，可以通过集群的镜像队列进行冗余。

memcached vs redis

发表于 2020-07-18 分类于 web开发

memcached vs redis

功能

memcached提供的功能是Redis的提供的功能的子集，Redis可以提供更丰富的数据类型及功能。

缓存大小

当缓存相对较小且数据为静态数据时，使用memcached是一个不错的选择，因为其存储的元数据消耗更少的内存。

当数据是动态的时候，memcached的内存管理效率下降快，内存会变成碎片，在数据较大的时候伴随着数据的序列化，需要更多的空间进行存储。

伸缩性

memcached是多线程，单机数据共享的（只能扩展机器，不支持不同主机间进行共享），因此可以通过增加主机进行扩展。

redis是单线程，可以通过对集群进行配置进行扩展，但集群扩展的操作相对复杂。

Redis如何动态扩容

场景描述：生产环境Redis内存爆满，如何在不重启Redis服务器的情况动态扩容？

单机模式：在单机模式下，可以通过redis客户端查看redis服务可用的最大内存，如果最大内存远未到达机器上限，则可以通过客户端进行配置而无需重启。

# 当前最大值为1G
config get maxmemory 1073741824

# 设置最大值为3G
config set maxmemory 3221225472

集群模式：使用redis-trib.rb工具向集群进行动态扩容，增加节点的顺序应该是先增加Master节点，然后再增加Slave节点。

# 增加新节点到集群中
ruby redis-trib.rb add-node newhost:newport clusterhost:clusterport

节点新增到集群后，新节点不会有任何数据，因为没有分配任何slot，后续需要手动分配，分配的方法为从已有的master节点进行分配，命令如下所示：

ruby redis-trib.rb reshard cluster-master-host:port

redis-cli登录从节点，分配新加入节点的主从关系：

cluster replicate 主节点ID

至此完成Redis集群的动态扩容。
